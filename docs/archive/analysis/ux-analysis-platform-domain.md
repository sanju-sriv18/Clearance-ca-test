## PART 1: Platform Operational Accuracy

### 1. Control Tower (Screenshots 01, 11)

**What an ops director actually needs**: A real-time nerve center showing clearance pipeline health, financial exposure, and exception queue prioritized by cost impact.

**Assessment**: The Control Tower is the strongest screen. It surfaces real operational data:

| Metric | Realistic? | Notes |
|--------|-----------|-------|
| Entries (Month) / Cleared | Real | Aggregated from `dashboard_aggregator.py`, queries actual `Shipment` table |
| STP Rate 87.3% | Realistic | Config default `stp_rate: 0.87` aligns with CBP's published ~87% STP rate |
| Avg Clear Time 2.4 hrs | Partially | Value is hardcoded fallback when no data exists (line 53 of `dashboard_aggregator.py`), not calculated from actual event timestamps |
| Hold Rate 3.8% | Realistic | Matches industry range (3-5%); dynamically calculated from held shipments |
| Duty Collected $384M / $99M | Real | Summed from `financials["predicted_duty"]` JSONB column |
| FTA Savings $12.4M / $3.17M | Questionable | Source of FTA savings calculation not found in dashboard aggregator — likely a residual default |
| Active Alerts (UFLPA, DPS/SDN, PGA, Classification) | Real | Exception counts from live shipment data |

**What's good**:
- Live SSE streaming via `useDashboardStream` + Redis pub/sub
- Corridor matrix shows real hold rates per trade lane (e.g., CN→US 5.2% vs MX→US 1.8%)
- Scrolling alert ticker shows actual demurrage accruing events with real dollar amounts
- "Your Attention Required" panel shows actual held shipments with dwell time and cost exposure

**What an expert would flag**:
- **P2**: "FTA Savings" metric lacks verifiable sourcing — the MX→US corridor shows 82% FTA utilization but the simulation has no FTA certification engine; USMCA claims are only validated at the document level with a 3% issue rate
- **P2**: Avg clear time `2.4 hrs` appears static across both screenshots (01 and 11), even though screenshot 11 shows a dramatically different volume (11.6K vs 47.8K entries). This confirms it's a hardcoded fallback, not computed from clearance event deltas
- **P3**: No cost-of-delay calculation visible — an ops director needs to see "this hold is costing $X/day in demurrage + storage + missed delivery windows"
- **P3**: Missing SLA tracking — percentage of entries cleared within 24/48/72 hours
- **P4**: The industry context footer ("$2.1T at risk, 72% entry errors industry-wide") is static marketing copy, not derived from portfolio data

**Severity**: P2 — Core metrics mix real and synthetic data without indication

## PART 2: Simulation Realism Audit

### Actor-by-Actor Assessment

#### 1. ShipperActor (`actors/shipper.py`) — REALISTIC

**Strengths**:
- Poisson arrival process with business-hour modulation — statistically sound
- Seasonal volume multiplier via `calendar.py`
- Weighted corridor selection matching real trade volume distribution (CN 25%, MX 14%, DE 10%)
- Misclassification injection (7%) and value error injection (3%) — exercising downstream compliance
- Multi-modal routing engine generates realistic waypoints and regulatory touchpoints
- Multi-product orders with 3-5 line items (30% of shipments) — realistic commercial mix

**Issues**:
- **P1**: **Tariff duties use hardcoded corridor rates, not the actual tariff engine.** Line 198: `predicted_duty = round(declared_value * corridor.avg_duty_rate, 2)`. The `avg_duty_rate` is a single flat number per corridor (CN→US: 0.384, MX→US: 0.012). In reality, duty depends on the HTS code, not the corridor. A semiconductor chip (8542.31.0000) from China has a very different duty rate than cotton textiles (6110.20.2075) from China. The flat-rate approach means all CN→US products get 38.4% duty, which is wrong for many product categories.
- **P3**: Weight/piece generation is uniform random within product ranges — real cargo follows a more peaked distribution (most shipments cluster near the mode, not uniform)
- **P4**: Company name selection is uniform random — real importers have power-law distribution (top 5 importers handle 60%+ of volume)

#### 2. CustomsActor (`actors/customs.py`) — PARTIALLY REALISTIC

**Strengths**:
- STP/manual-review cascading probability model is well-structured
- Corridor-specific scrutiny multipliers (CN: 1.3x, VN/IN: 1.1x) — realistic
- Entity screening safety net checks `FLAGGED_SHIPPER_NAMES`
- ADD/CVD case matching by HS prefix + origin country — uses real case numbers
- Mode-specific exam types (VACIS/NII for ocean, intensive/tailgate for air/ground)
- Backlog-aware slowdown when queue exceeds threshold

**Issues**:
- **P1**: **Customs decisions are 100% probabilistic dice rolls, not rule-based evaluation.** Line 258: `roll = self.rng.random()` determines the outcome regardless of shipment characteristics (except origin country). A shipment with a known misclassification (injected by ShipperActor) has the same probability of STP clearance as a clean shipment. The `misclassified` flag from the event payload is never read by the customs actor. This means the simulation produces random compliance outcomes rather than causal ones.
- **P2**: HS reclassification (line 469-486) randomly changes the last 2 digits and applies a random duty adjustment factor (0.8-1.3x). Real reclassification is based on product analysis, not random digit substitution.
- **P2**: Physical inspection pass rate (80%) is a flat probability — real inspection outcomes depend on what the exam finds (weight variance, product mismatch, concealment)

#### 3. ComplianceEngineActor (`actors/compliance_engine.py`) — DECORATIVE

**Issues**:
- **P0**: `run_real_engines` defaults to `False` in `ComplianceConfig` (line 64). The "real engine" delegation path (`_try_real_engines`) imports from `app.engines.classification`, `app.engines.tariff`, and `app.engines.compliance` — but these modules don't appear to exist in the codebase (import is wrapped in a broad `except Exception`). The actor generates synthetic analysis data with random confidence levels and shuffled duty program allocations.
- **P1**: Classification confidence is `random.choice(["HIGH", "HIGH", "HIGH", "MEDIUM"])` — 75% HIGH regardless of actual classification difficulty. A novel product from a high-risk corridor should have lower confidence.
- **P1**: Compliance status is 90% CLEAR / 10% REVIEW with no connection to actual risk factors. A shipment with a misclassified HS code, flagged origin, and UFLPA-risk product should have much higher review probability.
- **P2**: Duty line item generation picks 2-3 random programs from `DUTY_PROGRAMS` and normalizes shares. A real duty breakdown shows the specific programs applicable to that HS code/origin combination (MFN + Section 301 for CN electronics, just MFN for DE automotive parts).

#### 4. ResolutionActor (`actors/resolution.py`) — WAIT-AND-ROLL

**Issues**:
- **P1**: **Resolution is purely "wait N days + roll random outcome."** There is no investigation modeling. The actor waits 1-5 days (random), then rolls: 75% resolved, 10% escalated, 15% pending. The hold type (`uflpa`, `cf28_cf29`, `pga`, `inspection_failure`) has NO influence on resolution probability or timeline. In reality:
  - UFLPA holds average 6-12 months (not 1-5 days) and have ~15% release rate (not 75%)
  - CF-28 responses have a 30-day deadline (correctly modeled in CBPAuthorityActor but ignored here)
  - PGA holds depend on the agency: FDA averages 5 days, EPA 3 days (correctly modeled in PGAActor)
  - Entity screening holds may never resolve without evidence of exclusion

**Strength**: Transit hold resolution is correctly differentiated (returns to `in_transit` not `cleared`, shorter review periods for port authority vs EU sanctions).

#### 5. FinancialActor (`actors/financial.py`) — REALISTIC

**Strengths**:
- **MPF calculation is correct**: rate 0.3464%, min $29.66, max $614.35 — matches current CBP rates
- **HMF calculation is correct**: 0.125% on ocean imports only — matches current CBP rates
- **Broker fee tiers**: simple ($150) → standard ($275) → complex ($450) → critical ($650) + PGA surcharges ($175/agency) — realistic market rates
- **Bond types**: continuous (0.5% of annual duty, $50K min) and single-entry (1% of value, $100 min) — correct
- **Exam fees**: VACIS $350, intensive $850, tailgate $425 — realistic terminal charges
- **Payment methods**: ACH 60%, wire 20%, PMS 15%, check 5% — realistic distribution

**Issues**:
- **P2**: Bond premium calculation doesn't account for importer bond sufficiency — continuous bond amounts should be based on 10% of previous year's duty + tax + fees, not just a rate
- **P3**: No duty payment deadline tracking (10 days after entry for estimated duties, 10 days after liquidation for final)

#### 6. CBPAuthorityActor (`actors/cbp_authority.py`) — WELL-MODELED

**Strengths**:
- Risk-based response probability adjustment — higher risk score shifts probabilities toward CF-28/CF-29/exam
- Risk scoring considers: origin country, value threshold, HS chapter, ADD/CVD exposure, first-time importer
- CF-28 questions are domain-correct (classification, valuation, origin, compliance categories)
- CF-29 notices include rate advance, value adjustment, origin change — the three real CF-29 scenarios
- CF-28 deadline tracking (30 days) with timeout escalation
- Exam scheduling with realistic timing (1-3 days appointment window)
- Protest success rate (40%) — realistic for CBP protests

**Issues**:
- **P2**: CF-28 response rate probabilities (72% accept, 14% CF-28, 6% CF-29, 5% exam, 3% reject) are reasonable but not differentiated by entry type (informal vs formal, consumption vs warehouse)
- **P3**: No Entry Summary 7501 lifecycle — accepted entries should proceed to liquidation in 314 days

#### 7. BrokerSimActor (`actors/broker_sim.py`) — EXCELLENT DEMO ENGINE

**Strengths**:
- 5 realistic broker profiles with specializations, license numbers, and license districts
- Specialization-based assignment (FDA specialist → pharma entries, textile specialist → UFLPA-risk entries)
- Protected broker (David Kim) with reserved work for manual demo — smart UX choice
- Progressive workflow: draft → checklist completion → pending_broker_approval → submitted
- Document creation (Commercial Invoice, Packing List, Bill of Lading) tied to checklist state
- CF-28 auto-response with realistic category-appropriate text

**Issues**:
- **P3**: No broker workload capacity enforcement — the `workload_capacity: 20` field is set but never checked
- **P3**: Entry number format (FFF-XXXXXXX-Y) is correct but the check digit is random rather than algorithmically computed per the CBP specification

#### 8. PGAActor (`actors/pga.py`) — REALISTIC

**Strengths**:
- Agency-specific review windows and approval rates (FDA 1-5d/92%, EPA 1-3d/95%, CPSC 1-2d/97%)

**Issues**:
- **P3**: NHTSA review uses CPSC parameters (same window, same rate) — should have its own profile
- **P3**: No FCC equipment authorization checks for electronics imports

#### 9. DemurrageActor (`actors/demurrage.py`) — REALISTIC

**Strengths**:

## PART 4: Recommendations

### Immediate Actions (Connect Simulation to Real Engines)

1. **Replace corridor flat-rate duties with HS-code-aware tariff calculation** — The `avg_duty_rate` on the corridor should be replaced with a lookup against the product's HS code. Even a simple chapter-level rate table (Chapter 85 from CN = 25% Section 301 + 20% IEEPA = 45%; Chapter 61 from MX = 0% USMCA-eligible) would dramatically improve realism.

2. **Make customs actor read the `misclassified` flag** — The ShipperActor already injects this flag in the event payload. The CustomsActor should reduce STP probability for misclassified shipments (e.g., 30% instead of 87%).

3. **Differentiate resolution timelines by hold type** — UFLPA: 30-180 days, 15% release. CF-28: 5-30 days, 80% release after response. PGA: per-agency timelines (already partially modeled). Inspection failure: 3-10 days, 60% release.

4. **Wire ComplianceDashboard to real simulation data** — Replace the `generateDemoEntries()` function with an API call to the existing `/api/simulation/dashboard` endpoint or the entries API.

5. **Surface cargo exceptions in Exception Queue** — The ExceptionsActor generates `exception_damage`, `exception_shortage`, etc. as events. These should appear in the Exception Resolution queue alongside compliance holds.

### Strategic Improvements

6. **Build a Simulation Control Panel UI** — The backend has full API support (`/api/simulation/start|stop|pause|resume|reset`, actor config updates, clock control) but no frontend for it. An ops director should be able to:
   - Seed specific test scenarios (UFLPA hold, ADD/CVD review, ISF late filing)
   - Adjust actor parameters in real-time (increase hold rate, trigger disruption)
   - Fast-forward time to see resolution outcomes

7. **Create a tariff engine bridge** — Connect the simulation's duty calculations to the same LLM-based tariff engine used by Product Analysis, ensuring consistent results across all surfaces.

8. **Implement continuous entity screening** — Replace the 3-name hardcoded list with periodic re-screening against an updateable database, generating alerts when a known importer appears on a newly updated list.
